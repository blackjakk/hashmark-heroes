## 3. The ball-handler model — how it works & how to extend

All in `play-animation.js`. **ON by default for the four gadget kinds**; the
standard run/pass animators are untouched (only gadget plays take the path, so
their render is byte-identical and the AI-only gate batteries are unaffected).

- `_bhSampleBall(segs, aT)` — **the primitive.** The ball is a token; a play is a
  timeline of `HELD` segments (`hold(aT) → {x,y}`, ball tracks a holder's hand) and
  `FLIGHT` segments (parabolic arc between fixed release/catch points).
- `_bhGadgetAnim(env)` — the renderer. Builds a per-gadget **spec**
  `{ skip, handlers(dp,aT), segs, spotAt(aT) }`, then per frame: field → scaffold
  (`_bhDrawOL` + `_bhDrawDefense` converging on `spotAt` + `_bhDrawDecoys`) → spec
  handlers → **depth-sorted draw** (push to a sink keyed by field-Y, paint
  back-to-front) → ball → banner.
- **A new gadget = a new spec** (ball timeline + handler positions). No new renderer
  code — fake-spike / Statue of Liberty are now cheap.
- Intercept is at the top of `buildAnimForPlay` right after `attachPlayerStyles`,
  and **before** the function's own `ctx`/`fieldState` declarations (TDZ) → it uses
  a LOCAL ctx + its own `_fs`.

## 5. MIGRATION PLAN — normal run/pass plays → ball-handler model

> The big remaining payoff: collapse the bespoke run/pass animators into the
> unified model and kill the carrier/ball bug family at the root. Treat as a
> **separate, multi-session project** — the run/pass animators are the most
> battle-tested, most-tuned code in the repo (most of `CLAUDE.md`'s Pending list is
> fixes embedded there). **Never rewrite in place; intercept → prove parity → flip,
> one play-kind at a time, gated each step.** (Same intercept pattern the gadgets
> use today.)

### Stage 0 — Foundations
- **Derive the timeline in the animator**, not the engine. The engine emits an
  outcome + tracks (`motion.tracks.{carrier,wrN,tackler}`, `throwT`, `dropDepth`,
  `yac`, `targetDepth`), not a ball-event timeline. Deriving keeps the audit gate's
  byte-identical guarantee untouched (no engine change).
- **General spec builders** `_bhSpecForPass(play, env)` / `_bhSpecForRun(play, env)`
  that read those fields → segments + cast. The gadgets already sample
  `motion.tracks.*` via `_bhWRSampler` — reuse it.
- **Port the GUARDS, not just the choreography.** The standard animators carry
  hard-won continuity guards — catch-frame SNAP cap, carrier step cap (≤70px/frame),
  OOB clamp (`FIELD.TOP-6`/`BOT+24`), parked-defender leg-cycle-only liveness. These
  are the *root* of half the Pending fixes; the new model must reproduce them or the
  bugs return. **This is the subtle, essential work.**

### Stage 1 — Pilot (lowest stakes)
- New flag `GC_BH_NORMAL` (default off). Pick a low-traffic kind first
  (kneel/spike, or a vanilla completion).
- Build `_bhSpecForPass` for a basic dropback completion (snap→QB→drop→throw→catch→
  YAC) with the full 22 from formation + tracks.
- **A/B vs the standard animator** (side-by-side screenshots): catch point, YAC
  endpoint, pose families, zero teleports.

### Stage 2 — Runs (riskiest)
- `_bhSpecForRun`: snap→QB→handoff→RB carry (carrier track)→tackle.
- Hard part: the evasion poses (juke/truck/spin/stiff/hurdle) + tackle choreography
  + the pile — the most tuning lives here. Runs are also the bulk of the **teleport
  battery** → gate every iteration.

### Stage 3 — Pass variants
- Sack (snap→QB→pressure→sack, no throw), INT (throw→DB catch→return), screen
  (short throw→big YAC), play-action (longer drop). Each = a different timeline.

### Stage 4 — Special teams (last / optional)
- FG/punt/kickoff already have their own animators (punt is full-22). Lowest priority.

### Risks & mitigations
- **Resurrecting fixed bugs** → every migrated kind passes the *full* gate suite;
  keep the standard animator as a per-kind kill-switch until parity holds over time.
- **Tight teleport budget (4/4 runaway)** → the new model must not add teleports;
  depth-sort + spot-convergence + the ported step/continuity caps are the defense.
- **Pose fidelity (27 families, 0 flags)** → specs must drive the same pose
  families; pose audit stays green.
- **Perf** → 22 players × 60fps through canvas2D→PIXI sprite queue; the gadgets
  proved the load, normal plays are the same.
- **Don't flip a kind's default** until parity-validated across many seeds.

**Bottom line:** real, high-value, but deep. One kind at a time, fully gated,
standard animator retained as the fallback. Budget generously.
